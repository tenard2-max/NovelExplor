\### 0.4 핵심 설계 원칙 (전 장 공통 적용)

1\. \*\*원본 보존\*\*: 사용자가 Import한 원본 TXT/MD는 절대 수정하지 않는다.

2\. \*\*AI는 제안만 한다\*\*: 인물/세계관/복선/타임라인 후보를 자동으로 생성하지만, 사용자 승인 전에는 어떤 DB에도 저장되지 않는다.

3\. \*\*No Backend\*\*: Python/Node/Express/Flask/Django/외부 DB 사용 금지. HTML/CSS/JS + IndexedDB + File System Access API만 사용한다.

4\. \*\*모든 데이터는 Workspace 안에서만\*\*: 절대경로 저장 금지, 외부 접근 금지.

5\. \*\*삭제는 소프트 삭제\*\*: 휴지통 이동 후 30일 뒤 자동 영구삭제, 그 전까지 Undo 가능.

\## 1. 프로젝트 개요



\### 1.1 제품명

Fantasy Foreshadow Tool (판타지 떡밥 회수 프로그램)



\### 1.2 목적

판타지 장르 웹소설/장편소설 집필 시, 작가가 심어둔 복선(떡밥)을 잊지 않고 등급별로 추적·회수할 수 있도록 지원하는 저작 지원 도구를 만든다. GPT 등 LLM은 대화 세션이 길어지면 앞서 설정한 복선을 잊는 문제가 있으므로, 이를 영구적으로 저장·검색 가능한 로컬 데이터베이스로 대체한다.



\### 1.3 산출물 구조 (NovelMD)

```

NovelMD/

├─ 00\_MASTER.md

├─ 01\_STORY\_BIBLE.md

├─ 02\_WORLD\_SETTING.md

├─ 03\_CHARACTER\_DB.md

├─ 04\_FORESHADOW\_DB.md

├─ 05\_TIMELINE.md

├─ EP001.md \~ EP999.md

└─ Novel\_Master.zip

```

`Novel\_Master.zip`은 Master.md, Story Bible, 세계관 설정집, 등장인물 DB, F\~SSS급 복선 DB, 타임라인, 에피소드 원고를 하나로 묶은 배포용 백업 패키지다. 이 프로그램의 목적은 \*\*새로 업로드되는 스토리 TXT를 기준으로 위 산출물을 지속적으로 최신화\*\*하는 것이다.



\### 1.4 플랫폼 제약

\- 배포: GitHub Pages (정적 호스팅)

\- 허용 기술: HTML5, CSS3, JavaScript(ES Modules), IndexedDB, LocalStorage, File System Access API, Canvas API

\- 금지 기술: Python, Node.js, Express, Flask, Django, Electron, 외부 데이터베이스, 서버 API 키가 필요한 강제 연동

\## 2. 이해관계자 및 사용자 요구



\### 2.1 사용자 유형



| 유형 | 목표 | 주요 기능 |

|---|---|---|

| 작가 | 집필 및 복선 관리 | 소설 작성, 복선 등록/수정/회수, 등장인물 수정, 세계관 관리 |

\### 2.2 핵심 기능 요구사항 (요약)



| 영역 | 요구 기능 |

|---|---|

| Foreshadow DB | 등급(F\~SSS), 상태(생성/진행/회수/폐기), 중요도, 예정·실제 회수 화수, 관련 인물/사건/장소 |

| Import | txt, md, json, zip |

| Export | Novel\_Master.zip, NovelMD, txt, md, json |



\### 2.3 비기능 요구사항 (요약)

\- 500MB 프로젝트, 복선 10,000개, 소설 100,000줄 지원

\- 검색 1초 이내, 저장 0.5초 이내, 자동저장 주기 5초

\- 삭제 시 휴지통 이동, Undo 지원, 버전관리, 자동백업



\### 2.4 AI 요구사항 (전 기능 공통)

\- 내용을 임의로 수정하지 않는다.

\- 복선/인물/세계관/타임라인 후보만 제안한다.

\- 사용자가 승인해야 실제 DB에 저장한다.

\- 원본 TXT를 절대 변경하지 않는다.



\### 2.5 향후 기능

관계도 그래프, 복선 그래프, AI 요약, AI 모순 검사, AI 등장인물 분석 (상세 로드맵은 19장 참조)

\## 5. 데이터 모델 요구사항



\### 5.1 저장 원칙

모든 데이터는 Workspace 내부 JSON으로 저장하며 외부 DB는 사용하지 않는다 (GitHub Pages 전제).



\### 5.2 엔티티 관계도

```

Project

&#x20;├── Story ── Episode ── Paragraph ── Sentence

&#x20;├── Character

&#x20;├── World

&#x20;├── Timeline

&#x20;└── Foreshadow

```



\### 5.3 스키마



\*\*Project\*\*

```json

{ "projectId":"", "title":"", "author":"", "createdAt":"", "updatedAt":"",

&#x20; "version":"1.0.0", "workspace":"", "language":"ko" }

```



\*\*Episode\*\*

```json

{ "episodeId":"", "title":"", "number":1, "summary":"", "textFile":"",

&#x20; "createdAt":"", "updatedAt":"" }

```



\*\*Character\*\*

```json

{ "characterId":"", "name":"", "alias":"", "race":"", "gender":"", "age":0,

&#x20; "occupation":"", "description":"", "ability":\[], "firstEpisode":1,

&#x20; "lastEpisode":999, "status":"Alive" }

```



\*\*World\*\*

```json

{ "worldId":"", "category":"Kingdom", "name":"", "description":"",

&#x20; "parentId":"", "relatedCharacters":\[] }

```



\*\*Timeline\*\*

\## 6. Workspace 및 파일관리 요구사항



\### 6.1 목적

Workspace는 모든 프로젝트 데이터의 루트이며, 모든 읽기/쓰기는 Workspace 내부에서만 수행한다. 외부에는 어떤 파일도 생성·수정하지 않는다.



\### 6.2 구조

```

FantasyProject/

&#x20; Project.json

&#x20; NovelMD/  Story/  Character/  World/  Timeline/  Foreshadow/

&#x20; Relationship/  Dictionary/  Graph/

&#x20; Import/  Export/  Backup/  AutoSave/  Cache/  Temp/  Log/

```



\### 6.3 NovelMD 구조

`00\_MASTER.md, 01\_STORY\_BIBLE.md, 02\_WORLD\_SETTING.md, 03\_CHARACTER\_DB.md, 04\_FORESHADOW\_DB.md, 05\_TIMELINE.md, EP001.md\~EP999.md`



\### 6.4 Export 구조

`Novel\_Master.zip, Story.txt, Story.md, Story.json, Project\_Backup.zip`



\### 6.5 Backup 구조

`Backup/YYYY-MM-DD/Auto\_001.zip, Auto\_002.zip, Manual\_001.zip`



\### 6.6 Import 규칙

지원 파일: TXT, MD, JSON, ZIP. 압축 해제는 Workspace 내부 `Temp/`에서 수행한다.



\### 6.7 저장 절차

`Dirty Flag 확인 → JSON 저장 → Markdown 저장 → Backup 생성 → Version 증가`

기본 5초, 사용자가 1\~300초로 변경 가능, 자동 저장 시 UI를 차단하지 않는다.



\### 6.9 파일/파일명 규칙

\- 신규 파일 생성 위치: Story/Character/World/Timeline/Foreshadow/Relationship/Dictionary

\- 파일명은 UTF-8 사용

\- Episode: `EP001.md`, Character: `CHR\_0001.json`, Foreshadow: `FS\_0001.json`, Timeline: `TL\_0001.json`, World: `WD\_0001.json`



\### 6.10 삭제 규칙

즉시 삭제하지 않는다 → Recycle Bin 이동 → 30일 후 자동 삭제 → 그 전까지 Undo 가능



\### 6.11 Lock 규칙

동시에 하나의 파일만 수정 가능, 읽기는 다중 허용, 저장 중에는 Lock을 건다.



\### 6.12 캐시/Temp 규칙

\- Cache: AI 분석 결과, 검색 인덱스, Graph Cache, Thumbnail — 프로그램 종료 시 재생성 가능해야 함

\- Temp: 압축 해제, Import 작업, AI 임시 파일 — 종료 시 자동 삭제



\### 6.13 버전 파일

```json

```

저장 시 Patch 증가, Export 시 Build 증가



\### 6.14 로그 규칙

`Save.log, Error.log, Import.log, AI.log` — 최근 100개만 유지



\### 6.15 Workspace 유효성 검사

열기 시 `Project.json/NovelMD/Story/Export/Backup` 존재 여부 확인, 없으면 자동 생성



\### 6.16 보안 규칙

Workspace 외부 접근 금지, 절대경로 저장 금지(상대경로만), 실행 파일(.exe, .bat 등) Import 금지



\### 6.17 운영 원칙

모든 데이터는 Workspace 내부에서만 관리, 원본 Story는 항상 보존, AI는 원본 파일을 직접 수정하지 않음, 모든 변경은 사용자 승인 후 반영, Workspace는 다른 PC로 복사해 동일 프로젝트를 복원할 수 있어야 한다.

\## 7. Story Import 요구사항



\### 7.1 목적

외부 소설을 Workspace로 가져오되, Import 과정에서 원본을 절대 수정하지 않는다. 모든 분석은 복사본/메모리 상에서 수행한다.



\### 7.2 지원 형식

TXT, Markdown(MD), JSON, `Novel\_Master.zip`, `NovelMD.zip`, 일반 ZIP



\### 7.3 Import 흐름

`파일 선택 → 형식 검사 → Encoding 검사 → 압축 해제(필요 시) → Story Parser → Story Analyzer → Workspace 생성 → Story 저장 → Story Bible 생성 → Character 생성(후보) → Timeline 생성(후보) → Foreshadow 후보 생성 → 사용자 승인 → 최종 저장`



\### 7.4 인코딩

UTF-8, UTF-8 BOM, UTF-16, EUC-KR, CP949 지원. 자동 감지 실패 시 사용자가 선택한다.



\### 7.5 TXT/MD/NovelMD Import

\- TXT는 `Story/Original/EP001.txt` 형태로 원본 그대로 저장(수정 금지)

\- MD는 Heading/Code Block/Table/Quote/List 지원, 이미지는 링크만 유지

\- NovelMD는 `00\_MASTER.md \~ 05\_TIMELINE.md, EP001\~EP999.md` 구조를 자동 인식



\### 7.6 ZIP Import

`Temp/`에 압축 해제 → 구조 검사 → Workspace로 이동 → Temp 삭제



\### 7.7 자동 분석 항목

\- Story Parser: Chapter/Episode/Paragraph/Sentence/Line Number/Word Count

\- Character Detection: 이름/별명/호칭/등장횟수/첫등장/마지막등장 (신뢰도 낮으면 후보만 제안)

\- World Detection: 국가/도시/지역/마법/종족/용어

\- Foreshadow Detection: 복선 후보/떡밥/회수 문장/암시/반복 키워드 (자동 저장하지 않음)

\- Timeline Detection: 사건/날짜/순서/회상/플래시백/시간이동



\### 7.8 검증

중복 Episode, 깨진 링크, 잘못된 JSON, 손상된 ZIP, 누락 파일을 필수로 검사한다.



\### 7.9 중복 처리

동일 Episode 존재 시 덮어쓰기 또는 새 번호 생성 중 사용자가 직접 선택한다.



\### 7.10 오류 처리 및 로그

오류 발생 시 Import를 중단하고 원본을 유지하며 복구 가능해야 한다. `Import.log`에 시간/파일명/성공여부/경고/오류를 기록한다.



\### 7.11 AI 승인

AI가 생성한 Character/World/Foreshadow/Timeline은 모두 사용자 승인 후 저장한다.



\### 7.12 성능 목표

100MB TXT 기준 5초 이내 분석 시작, 500MB 프로젝트 Import 가능, 1000 Episode·100,000줄 지원



\### 7.13 향후 확장

DOCX, PDF, EPUB, 웹소설 플랫폼, Google Docs, Git Repository, Cloud Import

\## 9. Foreshadow Engine 요구사항



\### 9.1 목적

복선의 생성부터 회수까지 전 과정을 관리하는 핵심 엔진. AI는 복선을 자동 분석하지만 \*\*절대 자동 등록하지 않으며\*\*, 모든 결과는 사용자 승인 후 반영된다.



\### 9.2 Engine Pipeline

```

Story Import → Story Parser → Character Analyzer → Keyword Analyzer

→ Context Analyzer → Foreshadow Candidate Generator → Contradiction Checker

→ Timeline Linker → Story Bible Sync → User Approval → Database Save

```



\### 9.3 복선 Life Cycle

```

NEW → CANDIDATE → APPROVED → ACTIVE → EXPECTED → RESOLVED → ARCHIVED

&#x20;                                                      └──────→ CANCELLED

```

> 5장 데이터 모델의 단순 상태값(OPEN/PROGRESS/RESOLVED/CANCELLED)은 UI/저장 상의 요약 상태이며, 본 Life Cycle은 엔진 내부의 세부 처리 단계를 정의한다. UI에는 요약 상태로 매핑하여 표시한다 (`NEW/CANDIDATE→OPEN`, `APPROVED/ACTIVE/EXPECTED→PROGRESS`, `RESOLVED/ARCHIVED→RESOLVED`, `CANCELLED→CANCELLED`).



\### 9.4 Engine Modules

```

ForeshadowEngine

&#x20;├─ Story Parser        ├─ Pattern Analyzer

&#x20;├─ Character Analyzer  ├─ Context Analyzer

&#x20;├─ World Analyzer      ├─ Contradiction Checker

&#x20;├─ Timeline Analyzer   ├─ Relationship Analyzer

&#x20;├─ Keyword Analyzer    ├─ Grade Calculator

&#x20;└─ Resolution Checker  └─ Story Bible Sync

```



\### 9.5 세부 분석기 요구사항



| 분석기 | 추출/분석 대상 |

|---|---|

| Story Parser | 문장, 문단, Chapter, Episode, Line, Word Count |

| Character Analyzer | 이름, 별칭, 호칭, 등장빈도, 공동등장, 사망여부, 상태변화 |

| World Analyzer | 국가, 도시, 던전, 신, 종족, 마법, 아이템, 조직 |

| Keyword Analyzer | 반복 키워드, 특수 용어, 상징, 색, 숫자, 이름, 장소, 마법, 유물 |

| Context Analyzer | 암시, 복선, 예언, 회상, 꿈, 독백, 의심, 감정변화 |

| Pattern Analyzer | 반복 문장/단어/사건/인물/장소/아이템 |

| Contradiction Checker | 설정 충돌, 이름 변경, 시간 오류, 사망 후 등장, 장소 충돌, 마법 충돌 |

| Resolution Checker | 복선 회수 여부 자동 검사 (OPEN→EXPECTED→RESOLVED 또는 MISSED) |

| Relationship Analyzer | Character↔Character↔World↔Timeline↔Foreshadow Graph 생성 |

| Story Bible Sync | 승인된 복선을 Story Bible/Character DB/Timeline/World DB에 자동 연결 |



\### 9.6 후보 생성 예시 흐름

`"붉은 반지" 등장 → 후보 생성 → Grade 계산 → 예상 회수 Episode 계산 → 승인 대기`



\### 9.7 Grade System

`F, D, C, B, A, S, SS, SSS` — 산정 기준: 출현 빈도, 중요도, 연결 수, 스토리 영향도



\### 9.8 Confidence(신뢰도) 기준



| 범위 | 처리 |

|---|---|

| 90% 이상 | 자동 추천(승인 대기 목록 상단 노출) |

| 70\~89% | 추천 |

| 50% 미만 | 무시 |



\### 9.9 AI 규칙

원본 TXT 수정 금지, 자동 저장 금지, 삭제 금지, 사용자 승인 필수



\### 9.10 성능 목표

1000 Episode, 10,000 복선, 100,000줄 기준 — 검색 1초 이내, 분석 시작 5초 이내



\### 9.11 Engine Event Flow (증분 분석)

`Story Changed → Dirty Flag → Re Analysis Queue → Incremental Analyze → Changed Candidate → Approval → Database Update → Graph Update → Timeline Update → UI Refresh`



\### 9.12 향후 확장

\## 10. Character DB 요구사항



\### 10.1 목적

Character

&#x20;├─ Metadata   ├─ Relationship

{ "characterId":"CHR0001", "name":"", "alias":\[], "race":"", "gender":"",

\## 14. Save / Load 요구사항 (완성)



> 원본 `015\_SAVE\_LOAD.md`도 공통 소개문만 있었으므로, 003(Save Flow), 004(자동저장 UX), 006(저장 절차), 008(자동저장 절차)을 근거로 완성한다.



\### 14.1 목적

사용자의 작업 손실을 방지하기 위해 자동 저장과 수동 저장을 모두 지원하고, 저장 실패 시에도 데이터 무결성을 유지한다.



\### 14.2 저장 트리거

\- 수동 저장: `Ctrl+S`, 메뉴 File > Save

\- 자동 저장: Dirty Flag 발생 후 설정된 간격(기본 5초, 1\~300초 범위) 경과 시

\- 이벤트 저장: 프로젝트 닫기, 브라우저 종료 감지(`beforeunload`) 시 즉시 저장 시도



\### 14.3 저장 절차 (003 §8, 006 §7 근거)

```

사용자 수정 → Dirty Flag → Auto Save Queue → IndexedDB 저장

→ Workspace(파일 시스템, 지원 브라우저) 저장 → Version 생성

```

저장은 JSON 저장 → Markdown 동기화 저장 → Backup 생성 → Version 증가 순으로 원자적으로 수행하며, 중간 실패 시 이전 상태로 롤백한다.



\### 14.4 UI 요구사항 (004 §17, 008 §14 근거)

\- 자동 저장은 \*\*UI를 차단하지 않는다\*\* (백그라운드 처리, 저장 중에도 편집 가능)

\- 상태바에 저장 상태(저장됨/저장 중/저장 실패/Dirty)를 실시간 표시

\- 저장 실패 시 원인과 재시도 방법을 다이얼로그로 안내



\### 14.5 불러오기(Load) 절차

`프로젝트 목록 조회 → 선택 → Workspace 유효성 검사(6.15) → Project.json 로드 → Story/Character/World/Timeline/Foreshadow 로드 → Graph 생성 → UI 갱신`



\### 14.6 데이터 무결성

\- 저장 실패 시 직전 정상 버전(Version snapshot)으로 자동 복원

\- JSON 파싱 오류 감지 시 손상 파일을 격리(`Temp/corrupted/`)하고 최근 백업으로 대체 제안

\- 모든 저장 시점마다 Version.json build 값을 검증하여 데이터 일관성 확인



\### 14.7 File System Access API 연동 (선택 지원)

브라우저가 지원하는 경우 사용자가 로컬 폴더를 Workspace로 직접 지정하여 IndexedDB와 이중으로 저장할 수 있게 한다(파일 탐색기에서 직접 확인 가능). 미지원 브라우저에서는 IndexedDB + 수동 Export/Import로 대체한다.



\### 14.8 성능 목표

저장 0.5초 이내, 자동저장 주기 5초, 500MB 프로젝트에서도 UI 프리즈 없이 저장



\### 14.9 향후 확장

Cloud Save(선택적 연동), 다중 기기 동기화, 저장 충돌 해결(Conflict Resolution) UI

\## 15. Versioning 요구사항 (완성)



> 원본 `016\_VERSIONING.md`도 공통 소개문만 있었으므로, 004(Undo/Redo), 005(Version 스키마), 006(Version 규칙), 011(History/Rollback)을 근거로 완성한다.



\### 15.1 목적

모든 변경 이력을 추적하여 실수로 인한 데이터 손실을 방지하고, 특정 시점으로 되돌릴 수 있게 한다.



\### 15.2 버전 번호 체계 (006 §15 근거)

```json

{ "major":1, "minor":0, "patch":0, "build":15 }

```

\- \*\*Patch\*\*: 저장(Save) 시마다 증가

\- \*\*Build\*\*: Export(내보내기) 시마다 증가

\- \*\*Minor/Major\*\*: 사용자가 명시적으로 "버전 태그"를 남길 때 수동 증가(예: "1부 완성", "2.0 개정판")



\### 15.3 버전 스냅샷

저장 시마다 `versions` 저장소에 스냅샷 메타데이터(버전 번호, 저장 시각, 코멘트)를 기록한다. 스냅샷 본문(실제 데이터)은 일정 개수(예: 최근 50개) 또는 일정 기간만 보관하고, 이후는 자동 정리(Prune)한다.



\### 15.4 Undo/Redo (004 §17, 008 §15 근거)

\- 문장/필드 단위로 최소 1000단계 Undo/Redo를 지원한다.

\- Undo/Redo는 세션 종료 시 초기화될 수 있으나, Version 스냅샷은 세션을 넘어 영구 보관된다.



\### 15.5 History(변경 이력) — 11장 근거 확장

Character/World/Foreshadow/Timeline 등 모든 주요 엔티티는 개별 `History` 로그(작성자, 수정 시각, 변경 내용)를 가지며 항목 단위 Rollback을 지원해야 한다.



\### 15.6 수동 버전 태깅

사용자가 특정 시점에 "버전 저장"을 실행하면 Minor 버전을 올리고 코멘트(예: "3장 탈고")를 남길 수 있다. 이 지점은 자동 Prune 대상에서 제외한다.



\### 15.7 복원(Restore) 절차

버전 목록에서 특정 시점을 선택 → 미리보기(변경 diff 요약) → 복원 확인 다이얼로그 → 복원 시 현재 상태는 자동으로 새 버전으로 백업 후 교체한다(데이터 손실 방지).



\### 15.8 Export 버전과의 관계

`Novel\_Master.zip` Export 시 Build 번호가 증가하며, Export 파일명에 버전이 포함되어야 한다(`Novel\_Master\_v1.0.0\_build15.zip`).



\### 15.9 검증

버전 번호 역행 금지, 스냅샷과 실제 데이터 불일치 시 경고, 손상된 스냅샷은 목록에서 별도 표시



\### 15.10 성능/용량 목표

스냅샷 보관 최대 50개(또는 500MB 프로젝트 기준 용량 제한), 복원 작업 3초 이내

\## 16. File Format 요구사항 (완성)



> 원본 `017\_FILE\_FORMAT.md`도 공통 소개문만 있었으므로, 001(NovelMD/Novel\_Master.zip 구조), 005(JSON 스키마), 006(파일명 규칙), 007(지원 형식)을 근거로 완성한다.



\### 16.1 목적

프로젝트가 사용하는 모든 파일 형식과 그 구조를 명확히 정의하여, Import/Export/버전관리 간 호환성을 보장한다.



\### 16.2 내부 저장 포맷 (IndexedDB 오브젝트 스토어)

`projects, episodes, characters, worlds, foreshadows, timeline, versions, candidates, trash` — 각 레코드는 5장에서 정의한 JSON 스키마를 그대로 따른다.



\### 16.3 NovelMD 포맷 (사람이 읽을 수 있는 배포/백업 포맷, 001 근거)

```

NovelMD/

&#x20; 00\_MASTER.md          # 프로젝트 개요, 목차

&#x20; 01\_STORY\_BIBLE.md     # 세계관 총괄 요약

&#x20; 02\_WORLD\_SETTING.md   # World DB 내보내기

&#x20; 03\_CHARACTER\_DB.md    # Character DB 내보내기

&#x20; 04\_FORESHADOW\_DB.md   # Foreshadow DB 내보내기 (등급별 표)

&#x20; 05\_TIMELINE.md        # Timeline 내보내기 (연대기순)

&#x20; EP001.md \~ EP999.md   # 에피소드 원문(Markdown)

```

각 `.md` 파일은 YAML Front Matter(또는 표 형식)로 메타데이터를 포함하고, 본문은 사람이 읽기 쉬운 표/목록 형태로 렌더링한다.



\### 16.4 Novel\_Master.zip 포맷 (001 근거)

```

Novel\_Master.zip

&#x20;├─ Master.md

&#x20;├─ StoryBible/

&#x20;├─ WorldSetting/

&#x20;├─ CharacterDB/

&#x20;├─ ForeshadowDB/        # F/D/C/B/A/S/SS/SSS 등급별 하위 폴더

&#x20;├─ Timeline/

&#x20;└─ Episodes/ (EP001\~EPnnn)

```

압축은 표준 ZIP(Deflate)을 사용하며, 파일명은 전부 UTF-8로 인코딩한다.



\### 16.5 원본 Story 파일 (007 §5 근거)

Import된 원본은 `Story/Original/EPnnn.txt`(또는 `.md`)에 \*\*읽기 전용으로 보존\*\*하며, 편집은 항상 사본(`Story/Working/EPnnn.md`)에서 이루어진다.



\### 16.6 개별 DB 파일 명명 규칙 (006 §10 근거)

| 엔티티 | 파일명 패턴 |

|---|---|

| Episode | `EP001.md` |

| Character | `CHR\_0001.json` |

| Foreshadow | `FS\_0001.json` |

| Timeline | `TL\_0001.json` |

| World | `WD\_0001.json` |



\### 16.7 인코딩 규칙 (007 §4 근거)

저장은 항상 UTF-8(BOM 없음)을 사용한다. Import 시에는 UTF-8/UTF-8 BOM/UTF-16/EUC-KR/CP949를 인식하고, 저장 시 자동으로 UTF-8로 정규화한다.



\### 16.8 JSON 스키마 버전 관리

모든 JSON 파일 최상위에 `schemaVersion` 필드를 두어, 향후 스키마 변경 시 마이그레이션 스크립트가 이전 버전 파일을 자동 변환할 수 있게 한다.



\### 16.9 무결성 검사

\- ZIP: CRC 체크섬 검증, 손상 시 Import 중단

\- JSON: 스키마 유효성(필수 필드/타입) 검사, 실패 시 해당 레코드만 격리하고 나머지는 정상 처리(부분 실패 허용)



\### 16.10 호환성 원칙

파일 포맷 변경 시 이전 포맷과 최소 1개 상위 버전까지 하위 호환을 유지한다(Import 가능해야 함).



\### 16.11 향후 확장

EPUB 포맷 Export, PDF 조판 Export, 웹소설 플랫폼 업로드용 포맷(문피아/네이버시리즈 등) 변환기

\## 17. 내부 API 요구사항 (완성)



> 원본 `018\_API.md`도 공통 소개문만 있었으므로, 003(Application Layer의 Manager 구조), GitHub Pages 제약(서버 API 불가)을 근거로 완성한다. 여기서의 "API"는 \*\*외부 REST 서버 API가 아니라, 모듈 간 계약(Contract)이 되는 내부 JavaScript API 및 향후 플러그인 확장을 위한 인터페이스\*\*를 의미한다.



\### 17.1 목적

UI Layer가 Application Layer의 각 Manager를 일관된 방식으로 호출할 수 있도록 표준 인터페이스를 정의하고, 향후 Plugin System(LLM 연동 등)이 붙을 수 있는 확장 지점을 마련한다.



\### 17.2 설계 원칙

\- 서버 없음: 모든 API는 브라우저 내 JS 모듈 함수 호출이다(REST/GraphQL 서버 없음).

\- 각 Manager는 `create/read/update/delete/list/search` 형태의 일관된 메서드 시그니처를 제공한다.

\- 모든 쓰기 계열 메서드는 내부적으로 Dirty Flag를 세팅하고 Auto Save Queue에 등록해야 한다.

\- AI 관련 메서드는 반드시 `\*\_suggest()` 형태로 명명하여 "제안"과 "확정 저장"을 API 레벨에서도 구분한다(예: `foreshadow.suggestCandidates()` vs `foreshadow.approve(id)`).



\### 17.3 Manager별 표준 메서드 (예시 시그니처)



| Manager | 주요 메서드 |

|---|---|

| ProjectManager | `create()`, `open(id)`, `save()`, `list()`, `delete(id)`, `export(format)` |

| StoryManager | `importFile(file)`, `createEpisode()`, `updateEpisode(id, patch)`, `deleteEpisode(id)`, `getEpisode(id)` |

| CharacterManager | `list()`, `get(id)`, `create(data)`, `update(id, patch)`, `softDelete(id)`, `suggestCandidates(episodeId)`, `approve(candidateId)` |

| WorldManager | CharacterManager와 동일 패턴 |

| ForeshadowManager | 동일 패턴 + `checkResolution(id)`, `getGrade(id)` |

| TimelineManager | 동일 패턴 + `sortBy(mode)` (`date`/`episode`/`importance`) |

| SearchManager | `search(query, {scope, options})`, `reindex()` |

| GraphManager | `buildGraph(type)`, `getRelated(entityId)` |

| VersionManager | `snapshot(comment)`, `list()`, `restore(versionId)` |

| AutoSaveManager | `markDirty()`, `flush()`, `getStatus()` |



\### 17.4 이벤트 API (Pub/Sub)

UI 갱신을 위해 각 Manager는 표준 이벤트(`onCreate, onUpdate, onDelete, onCandidateReady, onSaveStateChange`)를 발행하고, UI Layer는 이를 구독하는 방식으로 결합도를 낮춘다.



\### 17.5 오류 규약

모든 API 호출은 실패 시 `{ ok:false, code, message }` 형태의 일관된 오류 객체를 반환하며, 성공 시 `{ ok:true, data }` 형태를 반환한다. UI는 이 규약에 따라 4.14의 오류 다이얼로그를 표시한다.



\### 17.6 Plugin/Extension API (확장 지점, 003 §11 · 009 §20 근거)

향후 LLM Plugin, Graph AI 등을 붙일 수 있도록 다음 확장 지점을 표준화한다.

\- `registerAnalyzer(name, analyzerFn)` — Foreshadow Engine 파이프라인에 분석기 추가

\- `registerExporter(format, exporterFn)` — Export 포맷 추가

\- `registerImporter(format, importerFn)` — Import 포맷 추가



플러그인은 반드시 "제안"만 반환해야 하며, 직접 DB에 쓰기 권한을 가지지 않는다(9.9/10.8 AI 규칙과 동일하게 적용).



\### 17.7 인증/키 관리 (선택 기능)

사용자가 자신의 LLM API 키(예: Anthropic/OpenAI)를 입력해 AI 분석기를 고도화할 수 있는 선택적 설정을 제공할 수 있다. 이 경우:

\- API 키는 브라우저 로컬(LocalStorage)에만 저장하고, 어떤 서버로도 전송하지 않는다.

\- 키가 없을 때는 규칙 기반(휴리스틱) 분석기가 기본으로 동작해야 한다(오프라인 100% 동작 보장).



\### 17.8 향후 확장

Webhook 형태의 외부 자동화 연동(Zapier류), 읽기 전용 공개 API(정적 JSON Export를 통한 열람용 웹뷰어 연동)

\## 18. 테스트 계획 (완성)



> 원본 `019\_TEST\_PLAN.md`도 공통 소개문만 있었으므로, 002(성능/무결성 요구), 004(오류 UI), 006(Lock/삭제 규칙), 007(Import 검증), 008(Undo/검증), 009(AI 규칙)을 근거로 완성한다.



\### 18.1 목적

모든 기능 요구사항과 비기능 요구사항이 구현되었는지 검증하기 위한 테스트 범위와 기준을 정의한다.



\### 18.2 테스트 레벨

| 레벨 | 대상 | 방법 |

|---|---|---|

| 단위 테스트 | 각 Manager의 CRUD 메서드, Parser/Analyzer 로직 | 자동화(JS 테스트 러너) |

| 통합 테스트 | Import→분석→승인→저장 전체 파이프라인 | 자동화 + 수동 시나리오 |

| UI 테스트 | Explorer/Editor/Inspector/Timeline/Graph 상호작용 | 수동 + E2E 스크립트 |

| 성능 테스트 | 대용량 프로젝트(500MB, 10,000 복선, 100,000줄) | 부하 스크립트 |

| 회귀 테스트 | 버전 업데이트 시 기존 프로젝트 호환성 | 자동화 |



\### 18.3 핵심 시나리오 테스트 (필수)

1\. \*\*원본 불변성\*\*: TXT Import 후 AI 분석·편집을 거쳐도 `Story/Original/`의 원본 바이트가 변경되지 않는지 검증한다.

2\. \*\*AI 승인 게이트\*\*: AI가 생성한 모든 후보(인물/세계관/복선/타임라인)가 사용자 승인 전에는 어떤 DB 테이블에도 `status='approved'`로 기록되지 않는지 검증한다.

3\. \*\*자동저장 신뢰성\*\*: 5초 자동저장 도중 강제로 탭을 닫아도(`beforeunload`) 마지막 Dirty 상태가 유실되지 않는지 검증한다.

4\. \*\*삭제/복구\*\*: 삭제된 항목이 즉시 사라지지 않고 휴지통에 보관되며, Undo 및 수동 복구가 가능한지 검증한다.

5\. \*\*모순 탐지\*\*: 사망 처리된 인물이 사망 이후 Episode에 재등장하는 경우 Contradiction Checker가 경고를 발생시키는지 검증한다.

6\. \*\*등급/상태 검증\*\*: Foreshadow의 Grade가 `F\~SSS` 외 값으로, Status가 4종 외 값으로 저장되지 않는지 검증한다.

7\. \*\*인코딩 처리\*\*: EUC-KR/CP949로 인코딩된 TXT 파일이 깨짐 없이 Import되는지 검증한다.

8\. \*\*검색 정확도/속도\*\*: 100,000줄 기준 전문검색이 1초 이내에 정확한 결과를 반환하는지 검증한다.

9\. \*\*버전 복원\*\*: 특정 버전으로 복원 시 복원 이전 상태가 자동 백업되어 데이터 손실이 없는지 검증한다.

10\. \*\*Export 무결성\*\*: `Novel\_Master.zip` 압축 해제 후 원본 데이터와 1:1 대응되는지(CRC/레코드 수 비교) 검증한다.



\### 18.4 성능 테스트 기준 (전 장 목표치 통합)



| 항목 | 목표 |

|---|---|

| 검색 응답 | 1초 이내 |

| 저장(수동) | 0.5초 이내 |

| 자동저장 주기 | 5초(설정 가능 1\~300초) |

| AI 분석 시작 | 5초 이내 |

| 프로젝트 최대 용량 | 500MB |

| 최대 복선 수 | 10,000개 |

| 최대 소설 분량 | 100,000줄 / 1,000 Episode |

| Undo 단계 | 최소 1,000단계 |



\### 18.5 오류/예외 테스트

저장 실패, Workspace 없음, JSON 손상, ZIP 손상, 중복 Episode, 빈 제목 저장 시도 각각에 대해 4.14의 오류 UI가 원인과 복구 방법을 정확히 안내하는지 확인한다.

키보드만으로 전체 기능(메뉴, Explorer, Editor, Inspector, 다이얼로그) 접근이 가능한지, 스크린 리더가 주요 라벨을 읽는지 확인한다.



\### 18.7 브라우저/환경 호환성

GitHub Pages 배포 기준 최신 Chrome/Edge/Firefox/Safari에서 동작을 확인하고, File System Access API 미지원 브라우저(Safari/Firefox)에서는 IndexedDB+수동 Export로 정상 대체되는지 확인한다.

\### 18.9 테스트 완료 기준(Exit Criteria)

1\. \*\*데이터 무결성 > 기능 다양성\*\*: 원본 보존, AI 승인 게이트, 자동저장 신뢰성이 최우선.

2\. \*\*오프라인 100% 동작\*\*: 어떤 Phase에서도 API 키 없이 핵심 기능이 동작해야 한다.

3\. \*\*성능 목표 회귀 금지\*\*: 새 기능 추가 시 18.4 성능 기준을 재검증한다.

\## 21. 용어사전



| 용어 | 정의 |

|---|---|

| 떡밥 / 복선 (Foreshadow) | 이후 스토리에서 회수(설명·전개)될 것을 암시하는 설정, 사건, 대사, 아이템 등 |

| 회수 (Resolution) | 복선이 스토리 내에서 실제로 설명되거나 결과로 이어지는 것 |

| Grade | 복선의 중요도 등급(F\~SSS) |

| Candidate | AI가 제안했지만 아직 사용자 승인을 받지 않은 항목 |

| Dirty Flag | 저장되지 않은 변경사항이 있음을 나타내는 상태 플래그 |

| Story Bible | 세계관·인물·복선·타임라인을 총괄 요약한 참고 문서 |

| Workspace | 프로젝트의 모든 데이터가 저장되는 루트 폴더 |

| NovelMD | 사람이 읽기 쉬운 Markdown 기반 배포/백업 문서 세트 |

| Novel\_Master.zip | NovelMD 전체와 원고를 하나로 묶은 백업 패키지 |

| Contradiction Checker | 설정 충돌(시간 오류, 사망 후 등장 등)을 탐지하는 검사기 |
